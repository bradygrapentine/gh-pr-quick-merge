# Adversarial Review — `v1.1-blockers-plan.md`

**Reviewer:** Opus (orchestrator)
**Date:** 2026-04-30
**Subject:** `/Users/bradygrapentine/projects/gh-pr-quick-merge/docs/plans/v1.1-blockers-plan.md`
**Source-of-truth doc:** `docs/status-2026-05-01.md`

The plan is broadly sound — Phase 0 preflight, four-track Phase 1 fan-out, sequential baselines, and the Codex routing for Sentry are all defensible calls. But several places conflate "independent files" with "independent semantics," misroute work to the cheapest agent, miss real blockers from the status report, and embed at least three foot-guns that would surface as CI red or store-listing rejection.

Findings below, by axis.

---

## 1. Sequencing correctness

### 1.1 Phase 1 tracks are not as independent as the plan claims (line 35)

The plan says "Merge order: 1A → 1B → 1C → 1D" and calls them "trivially serial." That undersells the actual coupling:

- **1A (version bump)** touches `package-lock.json`. If 1B/1C/1D are spun up as parallel worktrees off the same base SHA, every other branch will rebase-conflict on the lockfile the moment 1A merges. This is the exact `feedback_npm_lockfile_macos_linux_drift.md` failure mode. The plan should explicitly land 1A first and have the other branches `git pull --rebase main` before push, OR keep 1A out of the parallel batch entirely.
- **1D (privacy URL)** edits `manifest.json`. **1A also edits `manifest.json`** (version field). These are the same file. Calling these "no file overlap" (line 26) is wrong. Sequence 1A before 1D, or have 1D rebase.
- **1B's grep sweep** could surface dead refs in `lib/qm-onboarding.js` that 1D is also editing. The plan acknowledges this in a parenthetical (line 35) but doesn't act on it.

**Recommendation:** make Phase 1 a 2-step funnel: (1A only) → (1B || 1C || 1D), each rebasing onto post-1A main. Or just do 1A inline in Phase 0.

### 1.2 Phase 2 (visual baselines) before Phase 3A (branch protection) is the wrong order (lines 40, 58)

This is the single biggest sequencing error. Per `feedback_ci_branch_protection_paths_filter.md`: branch protection with required checks that don't pass on every PR silently blocks merges. The plan turns on branch protection requiring "Vitest, Playwright, lint, typecheck" (line 58) — but at the moment 3A lands, **11 Playwright visual specs are red** (status report §2). Until baselines are committed and merged into `main`, every PR — including the baseline PR itself — will be unmergeable behind its own required check.

**Recommendation:** Phase 2 (baselines) MUST land on `main` before Phase 3A flips on required Playwright checks. Plan currently has them in correct numerical order but the prose at line 62 ("3A is sequential; 3B and 3C are independent of 3A") makes it sound like 3A could ship first. Add an explicit dependency edge: `3A blocked-by 2.3`.

### 1.3 Phase 3B (Dependabot) and Phase 3C (Sentry vendoring) have a soft dependency the plan inverts (line 59)

3B says "ignore Sentry until 3C ships." But if 3C lands first, 3B doesn't need that ignore — it can pin the vendored version. Either ship 3C first and drop the ignore from 3B, or ship 3B first with the ignore and remove it after 3C. The plan mentions the dependency but doesn't pick an order. Pick one.

### 1.4 Phase 4 (HostAdapter scrub) should run before Phase 2

If 4.1 picks option (a) — inline the abstraction — that touches `content.js` and `lib/hosts/github/*`, which the visual baselines almost certainly capture (PR-page actions, row widget). Generating baselines, then refactoring the code that produces them, will re-invalidate the baselines. **Phase 4 before Phase 2.** The plan has them reversed.

## 2. Agent routing

### 2.1 Ollama on the version bump (1A) is a mistake (line 30)

The plan calls 1A "Trivial" and routes to Ollama as "mechanical, fully verifiable." But:

- `package-lock.json` regeneration is non-trivial. Per `feedback_npm_lockfile_macos_linux_drift.md`, on macOS you have to do `rm -rf node_modules package-lock.json && npm install` to avoid dropping Linux optional deps that CI's `npm ci` will then fail on. Ollama will not know this — it's a project-specific gotcha not visible in the prompt unless the dispatcher pastes the memory in.
- Manifest version, package.json version, and any `version` references in CI YAML or extension code (e.g., a `__VERSION__` constant, a Sentry `release` tag) need to move in lockstep. The status report explicitly flags Sentry release tagging as half-shipped (status §4). Ollama searching for "0.4.0" won't catch a `release: ${VERSION}` template.

**Reroute:** Sonnet, with the lockfile regen recipe inlined. Or do it inline in Phase 0 since it's a 5-minute task and the integration cost dominates.

### 2.2 Codex for Sentry vendoring (3C) is right, but for the wrong reason (line 64)

The plan justifies Codex via "security/CSP/data-handling adversarial review built in." That's the *review* skill. For the *implementation* — vendoring an SDK, wiring DSN from `chrome.storage.sync`, MV3 CSP shenanigans — Codex isn't materially better than Opus and is worse than Opus at multi-file wiring. The right pattern: **Opus implements 3C, Codex reviews via the codex-adversarial-gate skill before merge.** This is exactly what `codex_review_is_a_sampler.md` describes.

Specific things Opus would catch that Codex-as-implementer might miss:
- MV3 CSP: `manifest.json` `content_security_policy.extension_pages` vs `sandbox`. The plan says "ensure manifest.json allows it under MV3 CSP" but a vendored Sentry bundle that uses `eval` or `Function()` will be silently dropped by MV3 — needs an `unsafe-eval`-free Sentry build, not just a CSP edit. Worth surfacing in the brief.
- `scripts/package.sh` has a `vendor:sentry` step (status report §4) that the plan never references. 3C should integrate with that, not invent a parallel path.

### 2.3 Sonnet for 2.1 (CI YAML) is fine, but 2.2/2.3 over-rotates to Opus (lines 47–48)

Committing baselines and re-running the suite is mechanical: download artifact → unzip → `git add test/e2e/__snapshots__/` → push. Sonnet handles this. Opus's "judgment on which baselines to keep vs regenerate" (line 48) is a real concern but only if there are pre-existing baselines that shouldn't be overwritten — status report §2 confirms there are none ("A snapshot doesn't exist… writing actual"). So there's nothing to judge; just commit them. Reroute to Sonnet.

### 2.4 1D (privacy URL) at Opus is overkill for routine multi-file wiring

The status report makes clear the data-use copy already exists (the PRD has a data-use section per the plan itself, line 33). Sonnet can paraphrase it and wire 4 files. Reserve Opus for 3A (shared remote state) and Phase 4 (architectural call). Reroute 1D to Sonnet with a "do not invent new data-handling claims" guardrail.

## 3. Missing tasks

The status report flags items the plan ignores or under-treats:

1. **Coverage gate not executed** (status §4: "Coverage gate exists but was not executed in this audit"). If branch protection (3A) requires a coverage check, the plan needs to either (a) drop coverage from the required-checks list or (b) add a Phase 0.4 to confirm the coverage gate currently passes. Otherwise 3A bricks merges.
2. **`scripts/package.sh` `vendor:sentry` integration** (status §4). The plan's 3C reinvents vendoring without referencing the existing script step. Either delete the script step or wire 3C through it.
3. **Sentry `release` tagging** (Epic 5, status §1). When 1A bumps to 1.1.0, any Sentry `release` tag template needs to align — otherwise crash reports group under wrong release.
4. **Epic 9 partial-landing risk** is more than HostAdapter (status §4: "HostAdapter scaffold introduces an abstraction that has only one consumer"). Status report also calls out `lib/hosts/github/pr-state.js` extraction (PR #41) as part of the same partial landing. Phase 4 only audits `lib/hosts/github/*` but the dependent code in `content.js` is what needs the inline-or-keep decision.
5. **`docs/snapshot-update.md` SOP** (status §2): the plan never references the existing baseline-update SOP that requires "human reviewer sign-off per diff." If 2.2 commits baselines without that sign-off, it bypasses an existing process control.
6. **CWS/AMO listing copy drafting** is gated on the user (5.2/5.3) but the plan doesn't pre-draft. Sonnet should draft listing copy in parallel with Phase 2/3 so the user has something to paste when they hit the dashboard. The plan's open-question #3 acknowledges this but doesn't schedule it.
7. **Coverage of v1.3 metadata + filters surfaces in baselines** — the new filter bar / badges (PRs #48–#52) need their own visual specs. Status §2 lists 5 missing snapshots in `list-badges.spec.ts`; the plan treats baselines as one bucket without confirming coverage breadth.
8. **AMO requires source-code submission** for any minified/vendored JS. Once 3C vendors `@sentry/browser` (a minified bundle), AMO submission needs a source archive + build instructions. The plan's 5.3 doesn't budget for this. Common AMO rejection reason.

## 4. Risk in the plan itself

### 4.1 Branch protection PR-locks itself (already noted §1.2)

3A's required-checks list (line 58) names "Playwright" — the very suite that's currently red. If 3A merges before Phase 2, the next PR (Phase 4 or Phase 5) cannot merge without admin override.

### 4.2 Baseline regen on a feature branch may not match main's CI

Phase 2.1 pushes "a dispatch-only GH Actions workflow" on a feature branch (line 46). If the workflow file lives only on the feature branch, `workflow_dispatch` won't see it until it's on `main` (GitHub only exposes dispatch-able workflows from the default branch by default). The plan needs to either (a) merge the workflow to main first via a tiny PR, then run dispatch, or (b) trigger via `pull_request` event on the feature branch and extract artifacts from that run. The current phrasing will fail at "Trigger the workflow" (2.2).

### 4.3 Sentry CSP can break the extension on first install

3C wires the DSN from `chrome.storage.sync` (line 60) but doesn't say what happens on first run when no DSN is set. A Sentry init() with an empty DSN throws or no-ops depending on version. If it throws inside the content script, the extension's PR-page injection breaks. Add an explicit "no DSN → don't init" guard to the brief.

### 4.4 Manifest 1.1.0 without store-listing alignment

Bumping `manifest.json` to 1.1.0 (1A) before the CWS listing copy is updated (5.2) creates a window where users on the dev channel get 1.1.0 with stale store listing. This is only a problem if the build is auto-published, which it isn't (5.2 is human-gated). But the version label in onboarding screen 1 / options "About" pane, if any, will say 1.1.0 against a 1.0 store listing. Worth a check.

### 4.5 `git rm --cached .playwright-user-data/` is destructive without a guard

1C says "if committed, `git rm --cached`" (line 32). This is fine for `.playwright-user-data/` but the plan provides no rollback if the directory turns out to contain something the user actually wanted (e.g., a saved auth profile for local development). Status §4 only says it's "committed-or-tracked-shaped" — uncertainty. Verify with `git ls-files` first; don't blind-rm.

### 4.6 Phase 5.4 release tag uses `release-orchestrator` skill — verify it's hooked up

The skill exists per memory (`sdlc_skill_set.md`) but the plan doesn't confirm it's project-aware. If `release-orchestrator` writes to `CHANGELOG.md` and one doesn't exist, the skill may scaffold one without asking — minor scope expansion. Pre-flight check.

## 5. Verification gaps

The plan says "tests green" without specifying which suite, in multiple places:

- **Phase 1 verification gate** (line 36): "`npm test && npm run typecheck && npm run lint`." This omits `npm run test:coverage` (status §2 flags coverage gate as not run) and `npx playwright test` (which is currently red and needs to *stay* tracked even if it's gated by Phase 2). State explicitly: "unit + typecheck + lint must pass; Playwright is allowed to be red until Phase 2.3."
- **Phase 2.3** (line 48): "confirm 30/30 pass (or all-non-skipped pass)." Status §2 says 9 passed + 10 skipped + 11 failed = 30 total. So "30/30" is wrong — it should be "19/19 non-skipped" or "all formerly-failing-due-to-missing-baseline now pass." Tighten the gate.
- **Phase 3 has no verification gate at all.** 3A flips remote settings (no test). 3B adds a YAML file (validate with `gh dependabot` or at least a YAML lint). 3C ships code (needs the full unit suite + a manual extension load test in Chrome to confirm Sentry init doesn't throw). Add explicit verification for each.
- **Phase 4** (line 75): "tests stay green" is vague. If 4.1 picks (a) and inlines the abstraction, the diff is large and crosses content scripts. Require unit + typecheck + lint + Playwright e2e (non-visual) + a manual smoke load.

**No rollback path is specified anywhere.** Worth at least a one-liner per phase: "if 3A bricks merges, revert via `gh api -X DELETE /repos/.../branches/main/protection`"; "if 3C breaks injection, ship a hotfix that no-ops `lib/qm-sentry.js`"; "if 1A bumps to 1.1.0 and we need to roll back, the package-lock change is the binding constraint."

## 6. Open-questions completeness (lines 132–138)

The four questions cover privacy host, Sentry default, listing copy, and HostAdapter ADR. Missing:

5. **Is the coverage gate currently passing?** The status report explicitly flags this as unrun (§4). Before 3A turns coverage into a required check, the user needs to either confirm it passes or remove it from the required list.
6. **Should `.playwright-user-data/` be `git rm --cached`'d, or is it gitignored already?** Status §4 says "committed-or-tracked-shaped" — uncertainty. The plan's 1C assumes "if committed" without confirming. One `git ls-files | grep playwright-user-data` resolves it, but it's a user-visible decision (especially if the directory has saved auth state).
7. **AMO source submission policy for the vendored Sentry SDK?** Mozilla requires submitting unminified source + build steps for any minified third-party JS shipped in the extension. Once 3C lands, 5.3 needs a build-script artifact. Confirm the user is OK preparing that, or pick a non-vendored telemetry path.
8. **Does Sentry telemetry fire from content scripts, the service worker, or both?** Affects the CSP rules (3C) and the data shape disclosed in the privacy doc (1D). The plan doesn't surface this and the answer determines whether `connect-src` for `*.sentry.io` needs to live in `host_permissions` or just CSP.
9. **CHANGELOG.md format?** 5.4 uses the release-orchestrator skill but doesn't say if there's an existing CHANGELOG to extend. Worth pinning down before the skill auto-scaffolds one.

## Summary — top 5 issues to fix before executing

1. **Phase 3A (branch protection) must be blocked behind Phase 2.3 (baselines on main)**, not just numerically later. As written, 3A will brick PR merges via its own required Playwright check.
2. **Phase 4 (HostAdapter) should run before Phase 2 (baselines)** if option (a) is picked — refactoring after baselines re-invalidates them.
3. **Reroute 3C implementation to Opus, with Codex as the adversarial reviewer** (codex-adversarial-gate skill). The current routing inverts what Codex is good at.
4. **Reroute 1A off Ollama.** The lockfile regen recipe + Sentry release tag coupling exceeds Ollama's reliable scope. Inline it in Phase 0 instead.
5. **Phase 1 file-ownership claim is wrong** — 1A and 1D both edit `manifest.json`; 1A's lockfile bump conflicts every parallel branch. Funnel 1A first, then fan out 1B/1C/1D.

Plus the missing AMO source-submission obligation (§3.8) and coverage-gate uncertainty (§3.1, §6.5) — both can sink the launch if discovered late.
