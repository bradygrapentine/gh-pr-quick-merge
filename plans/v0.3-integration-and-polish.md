Ship all seven v0.3 power-user features: commit-message templates, keyboard shortcuts, per-repo defaults v2, stale-PR awareness, popup polish, token/auth UX, and the integration glue + test suite that ties them together. After this epic, `content.js` wires all five `lib/` modules, the options page manages every user preference, and fixture-DOM integration tests cover the full row-injection path.

## Conflict Matrix

The three hardest-to-parallelize files are `content.js` and `options.html`/`options.js`.

| File | Stories touching it |
|---|---|
| `content.js` | QM-031, QM-034, QM-035, QM-039, **QM-045** |
| `options.html` / `options.js` | QM-033, QM-036, QM-037, QM-038, QM-040, QM-044 |
| `popup.html` / `popup.js` | QM-041, QM-042, QM-043 |
| `styles.css` | QM-035, QM-039 |
| `lib/templates.js` | QM-032 |
| `lib/import-export.js` (new) | QM-038 |
| `background.js` | QM-043 |

**Key insight:** QM-031, QM-034, and QM-039 each add independent wiring to `content.js` (templates invocation, keydown listener, stale badge). They could in principle be parallelized into separate worktrees and merged, but the merge surface is the same file with adjacent edits. The integration risk is real. Instead, QM-045 is the designated convergence commit: it lands all three wires at once, after each lib module has been individually proven correct in isolation (Phase 1). Phases 1 and 3 are where parallelism is safe.

---

## Dispatch Decision

**Do not dispatch parallel subagents for Phase 2.** The `content.js` stories (QM-031, QM-034, QM-039) are short enough individually (each S or M) that cherry-pick overhead exceeds parallelism gain. QM-045 absorbs them cleanly.

**Phase 1 (lib + storage) and Phase 3 (UI work) each have 2–3 independent tracks** — dispatch is worthwhile there provided file-touch boundaries hold (see table above).

Recommended subagent split for Phase 1: one agent on QM-032 (templates storage helpers) + QM-043 (token badge background logic), another on QM-037 (autocomplete — pure options). They share no files.

Recommended subagent split for Phase 3: one agent on popup stories (QM-041, QM-042), one on options stories (QM-033, QM-036, QM-040, QM-044, QM-038). `options.html`/`options.js` is owned by the options agent; `popup.*` is owned by the popup agent. No overlap.

## Wave Structure

### Phase 1 — Pure lib + storage helpers (parallel-safe)

These stories touch no shared runtime files.

**Track A (run in parallel with B):**

- **QM-032** — Add `saveTemplate(name, body, store)`, `loadTemplate(name, store)`, `listTemplates(store)`, `deleteTemplate(name, store)` to `lib/templates.js`. Storage key: `qm_templates` (sync). Dual CJS + `window.QM_TEMPLATES` export. Add 6+ tests to `test/templates.test.js`.
- **QM-043** — Token-rotation reminder. In `background.js`, on `chrome.runtime.onInstalled` and daily alarm, read `chrome.storage.local.tokenSavedAt` (epoch ms). If `Date.now() - tokenSavedAt > 30 * 86400 * 1000`, set `chrome.storage.local.tokenStale = true`. Add 4+ tests using a mock alarm/storage harness.

**Track B (run in parallel with A):**

- **QM-037** — Repo-name autocomplete in the options page "Add per-repo default" input. On focus, call `GET /user/repos?per_page=100` using the stored token; populate a `<datalist>` element linked to the owner/repo input. Graceful degradation if the token is absent (datalist stays empty, no error). No new lib file needed — inline in `options.js`. Add 3+ tests (mock fetch, verify datalist population).

Gate: all Phase 1 tests green, no regressions, before moving to Phase 2.

### Phase 2 — content.js integration (serial)

These stories must land in order because each one edits `content.js`. QM-045 is the convergence commit.

1. **QM-031** — Wire `lib/templates` into the merge action in `content.js`. On merge button click, before firing the API call, load the default template for `owner/repo` (falling back to the global default, then `"Squash and merge"`). Pass as the `commit_title` / `commit_message` body fields. This is additive — no existing branches removed.

2. **QM-034** — Attach a `keydown` listener in `content.js` using `lib/shortcuts`. On `DOMContentLoaded`, call `shortcuts.register(keymap, handler)` where `handler` receives an action string (`"merge-focused"`, `"bulk-merge"`, etc.) and dispatches to the existing action functions. Load the user's custom keymap from `chrome.storage.sync.qm_shortcuts` with fallback to `shortcuts.DEFAULTS`.

3. **QM-039** — Inject stale-PR badge into rows. In `injectRow`, after computing `pr`, call `stale.isStale(pr.updated_at, thresholdDays)`. If true, append a `<span class="qm-stale-badge">Stale</span>` inside the row. Read `thresholdDays` from `chrome.storage.sync.qm_stale_days` (default 14). Add CSS class `qm-stale-badge` to `styles.css`.

4. **QM-035** — Focus-ring + a11y for shortcut activation. When a shortcut fires, add `aria-pressed="true"` to the activated button for 200ms, then restore. Add `.qm-shortcut-active` CSS class with a visible focus ring in `styles.css`. Screen-reader announcement via `aria-live="polite"` region already present in the DOM (add it if missing). This story is intentionally small — its content.js touch is 2–3 lines alongside QM-034.

   > Note: QM-035 can be committed immediately after QM-034 in the same Phase 2 run since they share no semantic conflict.

5. **QM-045** — Wave 2 integration glue (closes QM-022). This commit does not add new logic; it audits the `content.js` edits from steps 1–4, verifies no state leaks between modules, confirms the shared `state` object has slots for `state.templates`, `state.keymap`, `state.staleDays`, and `state.repoDefaults` (already wired from v0.2). Runs the full suite locally — must be green before merge.

Gate: `npm test` green with no skips; manual smoke-test against a fixture GitHub PR page.

### Phase 3 — UI work (parallel-safe after Phase 2 merges)

Two parallel tracks. Popup agent owns `popup.*`; options agent owns `options.*`.

**Track P — Popup agent (QM-041, QM-042, QM-043 UI side):**

- **QM-041** — Pinned-repo management UI in popup (closes QM-028). Add a "Manage" mode toggled by a button in the popup footer. In manage mode, each pinned-repo row gains a remove button (calls `chrome.storage.sync` to splice the entry). An "Add repo" input + confirm button appends a new `"owner/repo"` string to `pinnedRepos`. Validation: reject malformed slugs.

- **QM-042** — Manual refresh + empty/error states. Add a refresh icon button to the popup header; on click, re-run the fetch loop and re-render. Empty state: if `pinnedRepos` is empty, show the existing `EMPTY_STATE_HINT` + "Add your first repo" prompt. Error state: if any repo fetch fails (401, network), show a per-row error pill rather than crashing the whole list.

- **QM-044 UI side** — Multi-account hint. If `tokenStale === true` (set by QM-043 background logic), show a yellow banner in the popup: "Your token is 30+ days old — consider rotating it." Link opens options page. (QM-044 in the options page is owned by the options track.)

**Track O — Options agent (QM-033, QM-036, QM-038, QM-040, QM-044):**

- **QM-033** — Template editor UI in options page (closes QM-016). New section "Commit message templates". Lists saved templates (name + body preview). "New template" opens an inline form: name field, textarea, Save/Cancel. Uses `saveTemplate` / `listTemplates` / `deleteTemplate` from QM-032.

- **QM-036** — Shortcut customization UI in options page. New section "Keyboard shortcuts". Table of action → current binding → "Edit" button. Edit mode: key-capture input (listen for `keydown`, record the combo, show it). Save writes to `chrome.storage.sync.qm_shortcuts`. Reset button restores `shortcuts.DEFAULTS`.

- **QM-038** — Import/export for repo defaults + templates. New `lib/import-export.js` with `exportAll(store)` → JSON blob and `importAll(blob, store)` → validates schema + writes. Options page gets "Export settings" (triggers download) and "Import settings" (file-picker). Overwrites on import after a confirmation dialog.

- **QM-040** — Stale-threshold config UI. New field in the "Stale PRs" section of options: "Consider a PR stale after ___ days" (number input, min 1, max 365, default 14). On change, writes `chrome.storage.sync.qm_stale_days`; content.js picks it up via `onChanged`.

- **QM-044** — Multi-account hint in options page. Below the token input, show explanatory text: "Only one token is supported. To switch accounts, paste a token for the other account. Multiple simultaneous accounts are not supported." If `tokenStale === true`, additionally show the rotation warning in the options UI.

Gate for Track O: no regressions in `test/repo-defaults-integration.test.js`, `test/templates.test.js`, `test/shortcuts.test.js`; no existing option-section IDs renamed.

### Phase 4 — Integration test suite (serial, last)

**QM-046** — Fixture-DOM integration test suite (closes QM-010). Depends on QM-045 (all content.js wiring done).

Create `test/fixtures/` with at least three snapshot HTML files:
- `pr-list-clean.html` — a PR list row where `mergeable_state: "clean"`.
- `pr-list-stale.html` — a row where `updated_at` is 60 days ago.
- `pr-list-draft.html` — a draft PR row.

Create `test/content-integration.test.js`. Use `jsdom` (already a dev dependency via vitest) to load each fixture and run `injectRow` (extracted to a testable export or tested via module-level side effects with a mocked `chrome` global). Target: 12+ tests.

Suggested test cases:
1. Clean row → merge buttons injected, count correct.
2. Stale row → `.qm-stale-badge` present in DOM.
3. Non-stale row → `.qm-stale-badge` absent.
4. Draft row → merge buttons not injected (or disabled).
5. Template applied → button `data-commit-title` attribute set.
6. Shortcut keydown fires → action handler called.
7. Repo default present → `qm-btn-default` class on correct button.
8. Repo default absent → no `qm-btn-default` class.
9. `chrome.storage.onChanged` fires → rows re-styled without page reload.
10. Bulk select + shared default → dropdown pre-selected.
11. Bulk select + mixed defaults → dropdown left at user selection.
12. Row injection is idempotent (calling `injectRow` twice does not duplicate buttons).

## Open Questions

1. **`content.js` export pattern for testability (QM-046).** The file currently runs as a content script with side effects. Extracting `injectRow` as a named export (guarded by `if (typeof module !== 'undefined') module.exports = ...`) would allow jsdom testing without a full browser. Needs a decision before QM-046 starts — propose this as a micro-refactor at the top of Phase 4.

2. **`lib/import-export.js` schema versioning (QM-038).** The exported JSON needs a `version` field so future imports can migrate old blobs. Agree on version `1` now and bake the validator in from the start.

3. **Token-staleness threshold (QM-043).** 30 days is the plan's default. Confirm whether this should be user-configurable (adds a QM-040-style UI story) or hardcoded for v0.3.

4. **Options page section ordering.** Six new sections land in this epic. Agree on final order before Phase 3 starts to avoid constant re-ordering conflicts: recommended order — Token & auth, Per-repo defaults, Commit templates, Keyboard shortcuts, Stale PRs, Import/Export.

5. **Pinned-repo datalist source (QM-037).** `GET /user/repos` returns repos the authenticated user owns or has access to, not necessarily the ones they have PRs on. Consider caching the last-seen repo list from popup fetches as the autocomplete source instead, to avoid an extra API call.
