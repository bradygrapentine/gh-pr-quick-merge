# Plan — Row Actions: Update Branch, Merge-when-Green, Auto-Rebase (v0.4)

**Milestone:** v0.4
**Estimate:** XL (13 stories across 3 features)
**Dependencies:** Epic 1 integration glue — QM-045 must be merged so `content.js` has the row-injection pattern, token helper, and `behind_by` field exposed on each PR row.
**Parallel-safe with:** nothing outside this plan; F2.1/F2.6 and F2.2 are internally parallel in Wave 1 but converge on shared files in Wave 2.

---

## Goal

Add three row-level capabilities to every PR listed on `github.com/*/pulls`:

1. **Update Branch (F2.1)** — one-click sync of a stale branch via GitHub's `POST /pulls/:n/update-branch` API, with a configurable rebase-vs-merge strategy.
2. **Merge-when-Green (F2.2)** — queue a PR for automatic merge once all status checks pass; a background alarm polls and fires the merge when the coast is clear.
3. **Auto-Rebase before Merge (F2.6)** — optionally update the branch (rebase) automatically when the user hits the quick-merge button if the PR is behind by more than a configurable threshold, then proceed with the merge.

All three features inject UI into `content.js` rows and share `options.html` for configuration, making file-ownership coordination the primary execution risk.

## Internal wave structure

### Wave 1 — Pure libs (parallel, 3 tracks)

**Track A — QM-051: `lib/update-branch.js`**

Pure wrapper around `POST /repos/{owner}/{repo}/pulls/{pull_number}/update-branch`. Accepts `{ owner, repo, pullNumber, expectedHeadSha?, strategy }` where `strategy` is `"merge"` (default) or `"rebase"`. Returns `{ queued: true }` on 202, throws typed error on 4xx/5xx. Dual CJS + `window.QM_UPDATE_BRANCH` export.

Tests (target 8+):
1. Happy path — 202 → `{ queued: true }`
2. Strategy defaults to `"merge"` when omitted
3. Strategy `"rebase"` passes correct body
4. `expectedHeadSha` forwarded when provided, omitted when absent
5. 422 conflict → throws `UpdateConflictError`
6. 403 forbidden → throws `UpdateForbiddenError`
7. Network error propagates
8. Does not mutate input options object

**Track B — QM-054: `lib/merge-queue.js`**

Persistent queue over `chrome.storage.local` key `"mergeQueue"`. Shape: `Record<string, { owner, repo, pullNumber, addedAt, status: "watching"|"merged"|"failed" }>`. Exports: `enqueue(entry)`, `dequeue(key)`, `list()`, `updateStatus(key, status)`, `clear()`. All async, return Promises. Key = `"owner/repo#number"`.

Tests (target 10+):
1. `enqueue` persists entry with `status: "watching"`
2. `list()` returns all entries
3. `list()` on empty store returns `[]`
4. `dequeue` removes entry, returns it
5. `dequeue` on missing key returns `null`
6. `updateStatus` mutates only the status field
7. `updateStatus` on missing key is a no-op
8. `clear()` empties the store
9. Multiple enqueues don't overwrite each other
10. Key format matches `"owner/repo#number"` exactly

**Track C — QM-068: `lib/auto-rebase.js`**

Orchestrator. Exports:
- `shouldAutoRebase({ behindBy, threshold })` — returns `true` when `behindBy > 0 && behindBy >= threshold`. `threshold === 0` disables auto-rebase (returns `false`).
- `rebaseThenMerge({ owner, repo, pullNumber, expectedHeadSha, mergeStrategy, autoRebaseThreshold, onRebaseStart, onRebaseEnd })` — calls `shouldAutoRebase`; if true, calls `updateBranch` (importable from `lib/update-branch.js`), then calls the caller-supplied `mergeFn`; if false, calls `mergeFn` directly. Returns `{ rebased: boolean, merged: boolean }`.

Mock `lib/update-branch.js` in tests. Tests (target 8+):
1. `shouldAutoRebase` false when threshold 0
2. `shouldAutoRebase` false when `behindBy < threshold`
3. `shouldAutoRebase` true when `behindBy >= threshold` and threshold > 0
4. `rebaseThenMerge` skips rebase when `shouldAutoRebase` false
5. `rebaseThenMerge` calls `onRebaseStart` / `onRebaseEnd` when rebasing
6. `rebaseThenMerge` calls `mergeFn` after successful rebase
7. `rebaseThenMerge` propagates `updateBranch` error without calling `mergeFn`
8. Does not mutate input options

### Wave 2 — Content + Options integration (sequential)

**QM-055 — Background alarm + poller**

Register alarm `"mergeQueuePoller"` at 60-second period in `background.js` `onInstalled` and `onStartup`. Alarm handler: call `mergeQueue.list()`, for each `watching` entry fetch `GET /repos/{owner}/{repo}/pulls/{number}` + `GET /repos/{owner}/{repo}/commits/{sha}/check-runs`. If all check runs concluded green and `mergeable_state === "clean"`, call merge API, then `updateStatus(key, "merged")`. On API error set `"failed"`. Add `"alarms"` to `manifest.json` permissions array.

**QM-052 — Update button injection**

In `injectRowActions`: when `prData.behind_by > 0`, append an "Update" button. Click handler calls `updateBranch(...)` from `lib/update-branch.js`, shows inline spinner, on success reloads `behind_by` badge. Button disabled while in-flight. CSS class `qm-update-btn`.

**QM-056 — Queue state badge**

In `injectRowActions`: read `mergeQueue.list()` once on inject. If PR key present, render badge: 🟡 watching, ✅ merged, ❌ failed. Subscribe to `chrome.storage.onChanged` to re-render badge live without page reload.

**QM-057 — Cancel watch button**

When badge status is `"watching"`, render a "Cancel" button alongside the badge. Click calls `mergeQueue.dequeue(key)`, removes badge + button from DOM.

**QM-070 — Rebasing spinner**

In the quick-merge click path (existing), wrap the call with `rebaseThenMerge`. In `onRebaseStart` callback insert a `<span class="qm-rebasing">Rebasing…</span>` adjacent to the merge button. Remove on `onRebaseEnd`. CSS: subtle pulse animation, same visual weight as the existing merge spinner.

**QM-053 — Strategy toggle in options**

Add a `<section>` to `options.html`: "Update branch strategy" with a `<select>` containing `merge` (default) and `rebase` options. Save to `chrome.storage.sync` key `"updateBranchStrategy"`. Load on options open. `options.js` reads this key and exposes it to `content.js` via `chrome.storage.sync.get`.

**QM-069 — Auto-rebase threshold in options**

Add a `<section>` to `options.html` below QM-053's section: "Auto-rebase before merge — threshold (commits behind)". Number input, min 0, default 0 (disabled). 0 = off. Save to `chrome.storage.sync` key `"autoRebaseThreshold"`. Read in `content.js` before calling `rebaseThenMerge`.

## Risks

- GitHub's `update-branch` endpoint returns 202 (async) — the branch is not immediately updated. QM-052's button should poll or rely on a push event rather than assume instant readiness. Defer polling to a follow-up if needed; v0.4 shows a "queued" toast and refreshes `behind_by` after 3 seconds.
- Alarm poller fires every 60 seconds — acceptable latency for merge-when-green. Rate-limit exposure: one check-runs fetch per watched PR per minute. Cap queue at 10 entries in `enqueue` (reject 11th) to bound API usage.
- `content.js` multi-story edits are the highest integration risk. Assign one engineer (or one sequential agent) to Wave 2 entirely; do not parallelize these stories.
