# v1.0 Launch — Finish Plan

Concrete, copy-paste-ready steps for the remaining v1.0 launch blockers on
`bradygrapentine/gh-pr-quick-merge`. Cross-reference for the broader checklist:
[`docs/v1-launch-checklist.md`](../docs/v1-launch-checklist.md). This doc focuses on
the **how** for the five outstanding items only.

## Preamble — what's already done

- **Sentry SDK vendoring**: complete. `npm run vendor:sentry` works; `scripts/package.sh`
  invokes it when `SENTRY_DSN` is set; `background.js` conditionally loads the vendored
  bundle. Release-time activation knob is:
  ```bash
  SENTRY_DSN=<dsn> bash scripts/package.sh
  ```
  No further code work — DSN minted at release time only.

## Remaining blockers (5)

1. Branch protection on `main`
2. Dependabot security updates
3. CODEOWNERS file
4. Bot account `gh-pr-qm-bot` + `E2E_GH_TOKEN` secret
5. Store submissions — Chrome Web Store + AMO

---

## 1. Branch protection on `main`

**Why** — Prevents accidental force-push / direct-to-main and gates merges on the
three required CI jobs. Closes the "rolled bad code straight to main on a Friday"
incident class.

**Owner** — you (Brady).

**Estimated time** — 8 min.

### Steps

1. Open the rulesets UI:
   - Browser → `https://github.com/bradygrapentine/gh-pr-quick-merge/settings/rules`
   - Click **New ruleset → New branch ruleset**.
2. Fill the ruleset form with these literal values:
   - **Ruleset name**: `main protection`
   - **Enforcement status**: `Active`
   - **Bypass list**: leave empty (you do not need bypass; emergencies use a fresh PR)
   - **Target branches** → **Add target** → **Include default branch**
3. Under **Branch rules**, enable these checkboxes:
   - [x] **Restrict deletions**
   - [x] **Block force pushes**
   - [x] **Require a pull request before merging**
     - Required approvals: `0` (solo maintainer; flip to `1` when a co-maintainer is added)
     - [x] Dismiss stale pull request approvals when new commits are pushed
     - [x] Require conversation resolution before merging
   - [x] **Require status checks to pass**
     - [x] Require branches to be up to date before merging
     - **Add checks** — type each name and pick from the autocomplete list:
       - `test`
       - `manifest-lint`
       - `e2e`
   - [x] **Block force pushes** (already covered above; verify it's checked)
4. Click **Create**.

### Verification

```bash
gh api repos/bradygrapentine/gh-pr-quick-merge/rulesets \
  --jq '.[] | {name, target, enforcement}'
# Expect: {"name":"main protection","target":"branch","enforcement":"active"}

# Negative-path: open a throwaway PR with a deliberately failing test, then:
gh pr ready <num>
# Expect: GitHub blocks merge with "Required status check 'test' is failing"
```

## 2. Dependabot security updates

**Why** — Auto-PRs for known CVEs in `package.json` deps + GitHub Actions. Cheap
floor against supply-chain "you had a 9.8 in transitive X for 3 months" stories.

**Owner** — you.

**Estimated time** — 4 min.

### Steps

1. Toggle the repo-level features:
   - Browser → `https://github.com/bradygrapentine/gh-pr-quick-merge/settings/security_analysis`
   - Enable, in this order:
     - [x] **Dependency graph** (free, prerequisite for the rest)
     - [x] **Dependabot alerts**
     - [x] **Dependabot security updates**
     - [x] **Dependabot version updates** — *optional but recommended*; if enabled,
       it will prompt to commit a `dependabot.yml`. Use the config below.
2. (Only if you enabled version updates.) Commit `.github/dependabot.yml`:
   ```yaml
   version: 2
   updates:
     - package-ecosystem: npm
       directory: "/"
       schedule:
         interval: weekly
       open-pull-requests-limit: 5
       groups:
         dev-deps:
           dependency-type: development
     - package-ecosystem: github-actions
       directory: "/"
       schedule:
         interval: weekly
   ```

### Verification

```bash
gh api repos/bradygrapentine/gh-pr-quick-merge/vulnerability-alerts -i | head -1
# Expect: HTTP/2.0 204  (enabled). 404 = still off.

gh api repos/bradygrapentine/gh-pr-quick-merge/automated-security-fixes -i | head -1
# Expect: HTTP/2.0 204
```

## 3. CODEOWNERS file

**Why** — Auto-requests review from owner on every PR; required by some
branch-protection escalations later; documents the human accountable for
each path. Single-maintainer repo today, but cheap to seed.

**Owner** — you.

**Estimated time** — 5 min (commit + PR + merge).

### Steps

1. From a clean main checkout:
   ```bash
   git checkout main && git pull
   git checkout -b chore/codeowners
   mkdir -p .github
   cat > .github/CODEOWNERS <<'EOF'
   # Default owner for everything in this repo.
   *                       @bradygrapentine

   # CI and release tooling — extra eyes when these change.
   /.github/               @bradygrapentine
   /scripts/               @bradygrapentine

   # Extension entrypoints — manifest, background, content.
   /manifest.json          @bradygrapentine
   /background.js          @bradygrapentine
   /content/               @bradygrapentine
   EOF
   git add .github/CODEOWNERS
   git commit -m "chore: add CODEOWNERS"
   git push -u origin chore/codeowners
   gh pr create --fill
   ```
2. Wait for CI green, then merge via the standard squash flow.

### Verification

```bash
gh api repos/bradygrapentine/gh-pr-quick-merge/contents/.github/CODEOWNERS \
  --jq '.path'
# Expect: ".github/CODEOWNERS"

# Open a no-op PR — confirm @bradygrapentine appears under "Reviewers" automatically.
```

## 4. Bot account `gh-pr-qm-bot` + `E2E_GH_TOKEN` secret

**Why** — E2E tests that exercise the GitHub API need an isolated identity.
Running them under your personal PAT pollutes audit logs, risks rate-limit
collisions with normal dev work, and leaks scope (your personal PAT has way
more than `repo`+`workflow`).

**Owner** — **human-only, one-time** (this is you, Brady, but the steps require
real-human signup flows that the agent cannot script).

**Estimated time** — 25–35 min (mostly waiting on email confirmations + 2FA enrollment).

### Steps

#### a. Sign up the bot account

1. Open a private/incognito window so you don't clobber your main GitHub session.
2. Use a fresh email alias — recommended pattern:
   `grapentineb+gh-pr-qm-bot@gmail.com` (Gmail "+" aliasing routes to your inbox
   but registers as a distinct address with GitHub).
3. Browser → `https://github.com/signup`
   - Username: `gh-pr-qm-bot`
   - Email: the alias above
   - Password: generate fresh in 1Password / your manager — do **not** reuse.
4. Complete the email verification.
5. Enable 2FA immediately:
   - Settings → Password and authentication → **Enable two-factor authentication**
   - Use TOTP (1Password / Authy). Save recovery codes to your password manager.
6. (Optional but cheap) Set a profile note: bio "Automation account for
   gh-pr-quick-merge E2E tests. Owned by @bradygrapentine."

#### b. Mint a fine-scoped PAT (still inside the bot's session)

1. Settings → Developer settings → **Personal access tokens** → **Fine-grained tokens**
   → **Generate new token**.
2. Fill the form:
   - **Token name**: `gh-pr-quick-merge E2E`
   - **Expiration**: `90 days` (calendar a renewal)
   - **Resource owner**: `gh-pr-qm-bot`
   - **Repository access**: **Only select repositories** → add
     `bradygrapentine/gh-pr-quick-merge`
   - **Repository permissions**:
     - `Contents`: Read and write
     - `Pull requests`: Read and write
     - `Workflows`: Read and write
     - `Metadata`: Read (auto-enabled)
   - All other permissions: leave at **No access**.
3. Generate. Copy the `github_pat_...` value into your password manager
   immediately — GitHub will not show it again.

> Note: the user instructions said `repo` + `workflow` scopes, which is
> the **classic** PAT scope vocabulary. Fine-grained tokens express the same
> reach via the permissions list above and are strictly safer (repo-scoped, not
> account-wide). If you specifically need a classic PAT, the equivalent is:
> Settings → Developer settings → Tokens (classic) → Generate new (classic) →
> select `repo` + `workflow` only.

#### c. Invite the bot as a collaborator

Bot accounts cannot push to a repo they don't have access to. From your **main**
account:

```bash
gh api -X PUT repos/bradygrapentine/gh-pr-quick-merge/collaborators/gh-pr-qm-bot \
  -f permission=write
```

Then, from the bot's session, accept the invite at
`https://github.com/bradygrapentine/gh-pr-quick-merge/invitations`.

#### d. Store the PAT as an Actions secret

From your **main** account, on a shell where the PAT is in `$TOK`:

```bash
read -s TOK   # paste the github_pat_... value, press enter
gh secret set E2E_GH_TOKEN \
  -R bradygrapentine/gh-pr-quick-merge \
  --body "$TOK"
unset TOK
```

### Verification

```bash
gh secret list -R bradygrapentine/gh-pr-quick-merge | grep E2E_GH_TOKEN
# Expect: E2E_GH_TOKEN  Updated <today>

gh api repos/bradygrapentine/gh-pr-quick-merge/collaborators/gh-pr-qm-bot/permission \
  --jq .permission
# Expect: "write"

# Trigger or wait for the next E2E workflow run; check the e2e job authenticates
# as gh-pr-qm-bot (the `actor` field on test PRs it creates will read
# "gh-pr-qm-bot", not "bradygrapentine").
```

### Renewal checklist (90 days)

- [ ] Calendar reminder set for token expiry
- [ ] On expiry: re-mint with the same permissions, re-run `gh secret set E2E_GH_TOKEN`
- [ ] Confirm next E2E run is green

## 5. Store submissions — Chrome Web Store + AMO

**Why** — v1.0 isn't shipped until the extension is live in both stores.
Listing copy and review docs are already drafted; this section sequences the
upload flow.

**Owner** — you.

**Estimated time** — 90 min hands-on across both stores; **3–10 day review
latency** (CWS often <24 h, AMO 1–10 days for new listings).

### Cross-cutting prep (do once)

```bash
# Build the production zip + AMO source bundle.
SENTRY_DSN=<your release DSN> bash scripts/package.sh

# Expect outputs in dist/:
#   dist/gh-pr-quick-merge-<ver>.zip          (CWS upload)
#   dist/source-disclosure.zip                (AMO source bundle)
ls dist/
```

Re-read the listing review path before clicking submit:
[`docs/store-submission-guide.md`](../docs/store-submission-guide.md).

### 5a. Chrome Web Store (CWS)

**Estimated time** — 30 min upload + 0–24 h review.

#### Steps

1. Sign in to the Developer Dashboard:
   `https://chrome.google.com/webstore/devconsole/`
   (One-time $5 developer-account fee if you haven't paid it.)
2. **Add new item** → upload `dist/gh-pr-quick-merge-<ver>.zip`.
3. **Store listing** tab — paste content from `creative/store-copy/cws-listing.md`:
   - Detailed description
   - Short description
   - Category: `Developer Tools`
   - Language: `English`
4. **Graphic assets** — upload from `creative/store-copy/`:
   - 128×128 icon (already in repo `icons/icon-128.png`)
   - Screenshots per `creative/store-copy/screenshot-specs.md`
   - Promo tile per `creative/store-copy/promotional-tile-spec.md`
5. **Privacy practices** tab:
   - Single purpose: "One-click squash-merge for GitHub PRs from the PR list"
   - Permission justifications: copy from `docs/store-submission-guide.md`
   - Data usage certification: certify "no remote code", "no PII collected"
     (Sentry DSN is opt-in; document this on the privacy page).
6. **Distribution** tab:
   - Visibility: `Public`
   - Regions: `All regions`
   - **Rollout**: **Staged — 10%** (per the launch plan; ramp on day-3 if
     no Sentry crash spike).
7. Click **Submit for review**.

#### Verification

```bash
# Listing live (after review approves):
curl -sI "https://chrome.google.com/webstore/detail/<ext-id>" | head -1
# Expect: HTTP/2 200

# Confirm rollout %: visible in the dashboard under "Distribution"; cannot be
# fetched via API. Screenshot for the launch log.
```

### 5b. Mozilla Add-ons (AMO)

**Estimated time** — 45 min upload + 1–10 day review.

#### Steps

1. Sign in: `https://addons.mozilla.org/developers/`. Use the same account email
   you've used for Firefox sync if any; otherwise create a fresh AMO account.
2. **Submit a New Add-on** → choose **On this site** (listed).
3. Upload `dist/gh-pr-quick-merge-<ver>.zip` (the CWS-bound zip is the same
   artifact — Manifest V3 cross-store).
4. AMO will ask "Do you need to submit source code?" — answer **Yes** (the
   bundle includes minified vendored Sentry, so reviewers need the source to
   audit). Upload `dist/source-disclosure.zip`.
5. **Listing details** — paste from `creative/store-copy/amo-listing.md`:
   - Name, summary, description
   - Categories: `Developer Tools`
   - Tags: `github`, `pull-request`, `productivity`
   - Privacy policy URL: link to the privacy section in your repo README or a
     hosted page (must be reachable; a GitHub Pages section is fine).
6. **License**: `MIT` (or whatever ships in `LICENSE`).
7. Submit for review.

#### Verification

```bash
# Listing live (after review approves):
curl -sI "https://addons.mozilla.org/firefox/addon/gh-pr-quick-merge/" | head -1
# Expect: HTTP/2 200

# Pre-publication, the developer dashboard shows status. Reviewer feedback
# (if any) arrives via email to the AMO account.
```

#### If AMO review comes back with changes

- Address the reviewer note in code on a feature branch.
- Re-run `SENTRY_DSN=<dsn> bash scripts/package.sh`.
- Upload the new zip + new `source-disclosure.zip` to the same listing version.
- Reply to the reviewer in the same thread — do not open a new submission.
