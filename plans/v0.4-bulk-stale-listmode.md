# Plan — Bulk close+label, Stale-PR highlighting, List-endpoint mode (Epic 2: F2.3–F2.5)

**Milestone:** v0.4
**Stories:** QM-058–067
**Dependencies:** Epic 1 must be merged (content.js row injector stable, `lib/stale-pr.js` shipped, options page exists)
**Parallel-safe:** F2.3, F2.4, and F2.5 operate on largely disjoint files — parallel dispatch is viable if row-injector hooks are frozen before dispatch (see Dispatch Decision below)

---

## Goal

Ship three Epic 2 features against the v0.4 milestone:

- **F2.3 Bulk close + label** — extend the existing bulk-action bar so users can close selected PRs or apply a label to them in one click, with confirmation and per-row result feedback.
- **F2.4 Stale-PR row highlighting** — integrate the already-shipped `lib/stale-pr.js` engine into the row injector, add per-repo threshold configuration to the options page, and improve the stale badge with a tooltip and proper accessibility attributes. This goes beyond the basic badge from F1.4 (Epic 1).
- **F2.5 List-endpoint mode** — add a "fast mode" that fetches PRs from the cheaper `/repos/{owner}/{repo}/pulls` list endpoint instead of the per-PR detail endpoint, gated by a `listModeEnabled` storage flag and a toggle in the options page.

## TDD spec for `lib/bulk-ops.js` (QM-058)

Exports:

- `closePRs(repoFullName, prNumbers, token)` — calls `PATCH /repos/{owner}/{repo}/pulls/{number}` with `{ state: "closed" }` for each number. Returns `[{ number, ok: bool, error?: string }]`. Never throws — errors are captured per-PR.
- `labelPRs(repoFullName, prNumbers, labels, token)` — calls `POST /repos/{owner}/{repo}/issues/{number}/labels` for each number. Same result shape.
- `DEFAULT_CONFIRM_THRESHOLD` — constant `5` (number of PRs above which a second confirmation is required).

Dual CJS + `window.QM_BULK_OPS` export.

### Tests (target: 14+)

1. `closePRs` with one PR — issues one PATCH, returns `[{ number, ok: true }]`.
2. `closePRs` with three PRs — issues three PATCHes.
3. `closePRs` — API error on one PR → that entry has `ok: false, error: "<message>"`, others unaffected.
4. `closePRs` — empty `prNumbers` array → returns `[]` without any fetch calls.
5. `closePRs` — token passed in Authorization header.
6. `labelPRs` happy path — one PR, two labels.
7. `labelPRs` — multiple PRs, labels array sent for each.
8. `labelPRs` — API 422 (label not found) → captured per-PR, does not throw.
9. `labelPRs` — empty `prNumbers` → returns `[]`.
10. `labelPRs` — empty `labels` array → returns `[]` without any fetch calls.
11. Result array order matches `prNumbers` input order.
12. `closePRs` uses correct endpoint path (`/pulls/{n}`, not `/issues/{n}`).
13. `labelPRs` uses correct endpoint path (`/issues/{n}/labels`).
14. Network timeout (fetch rejects) captured as `ok: false`.

## TDD spec for `lib/list-mode.js` (QM-065)

Exports:

- `fetchPRList(repoFullName, token, options?)` — calls `GET /repos/{owner}/{repo}/pulls?state=open&per_page=100` (paginating if `Link` header present). Returns array of normalized PR objects: `{ number, title, draft, updatedAt, user, labels, mergeable_state: null }`. Note `mergeable_state` is always `null` — the list endpoint does not return it.
- `normalizePR(rawPR)` — pure mapper from GitHub list-endpoint object to normalized shape. Exported for unit testing.
- `LIST_MODE_FIELDS` — constant array of field names the list endpoint provides (documents what callers can rely on vs what requires detail endpoint).

### Tests (target: 10+)

1. `normalizePR` maps `updated_at` → `updatedAt` (camelCase).
2. `normalizePR` sets `mergeable_state: null` regardless of raw input.
3. `normalizePR` preserves `draft`, `number`, `title`, `user`, `labels`.
4. `fetchPRList` — single page, no `Link` header → returns normalized array.
5. `fetchPRList` — two pages via `Link: <...>; rel="next"` → concatenates both pages.
6. `fetchPRList` — empty list → `[]`.
7. `fetchPRList` — passes token in Authorization header.
8. `fetchPRList` — non-200 response → throws with status code in message.
9. `fetchPRList` — `per_page=100` in URL.
10. `LIST_MODE_FIELDS` does not include `mergeable_state`.

## Dispatch decision

Three features, largely disjoint file ownership — parallel dispatch is viable. Two conditions must hold before dispatching:

1. **Epic 1 is fully merged to main** — including F1.4's basic stale badge. Verify with `git log origin/main --oneline -10` and confirm the stale-badge commit is present.
2. **Row injector hook is stable** — `content.js`'s `injectRowExtras` signature is not being modified by any in-flight branch. Check `gh pr list --state open` before dispatch; if any open PR touches `content.js`, resolve or merge it first.

If both conditions hold, dispatch three parallel tracks:

- **Track A (F2.3):** owns `lib/bulk-ops.js`, `lib/label-picker.js`, `test/bulk-ops.test.js`, and the F2.3 edits to `content.js` + `content.css`. Stories run serially within the track: QM-058 → QM-059 → QM-060 → QM-061.
- **Track B (F2.4):** owns `lib/stale-pr.js` (read-only import), `test/` (F2.4 additions), and the F2.4 edits to `content.js` + `content.css`. Does NOT own `options.html`/`options.js`. Stories: QM-062 → QM-064 (QM-063 deferred to options integration step).
- **Track C (F2.5):** owns `lib/list-mode.js`, `test/list-mode.test.js`, the F2.5 edits to `content.js`, and all options page edits (`options.html`, `options.js`). Integrates QM-063 (F2.4 per-repo threshold UI) alongside QM-067 (fast mode toggle) in a single options-page pass. Stories: QM-065 → QM-066 → QM-067 (with QM-063 folded in).

**content.js touch points:** Track A edits the bulk-action bar section; Track B edits the row-injector section; Track C edits the fetch-path branch. These are structurally separate regions. If conflict arises on merge, Track A merges first (bulk bar), then Track B (row injector), then Track C (fetch path).

**Merge order:** A → B → C. Track C rebases last to absorb any minor `content.js` line-number drift from A and B.

If either condition above does not hold, fall back to serial implementation in the same order (A, then B, then C).

## Risks

- **content.js merge conflicts** — three tracks all touch the file. Mitigated by structural separation of edit regions and the A → B → C merge order. If Epic 1 leaves `content.js` in a fragmented state, serialize all three tracks instead of parallelizing.
- **GitHub API rate limits during bulk ops** — `closePRs` / `labelPRs` issue one request per PR. For large selections this can exhaust the 5000 req/hr limit quickly. v0.4 ships no rate-limit guard; log a follow-up issue.
- **List endpoint missing `mergeable_state`** — callers of `fetchPRList` that previously rendered a merge-readiness indicator will show nothing in fast mode. Options UI must make this trade-off visible to the user (tooltip or note on the toggle).
- **F2.4 / F1.4 badge duplication** — if F1.4 already injects a `<span class="qm-stale-badge">` and F2.4 also injects one, the row will show two badges. F2.4 implementation must check for an existing badge and update it in place rather than appending a new one.
