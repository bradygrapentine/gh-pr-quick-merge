# v1.2 Epic 10 — PR-Page Safety: Always-Visible Rebase Button

**Status:** draft
**Base SHA:** rebase onto `origin/main` at start (capture SHA in PR description)
**Total est:** ~3 eng-days (single author, single PR squash-merge)
**Stories:** QM-400 .. QM-410 (11 stories)

## Dependencies

- Epic 8 design tokens merged (action color tokens — green/yellow/blue/purple/red — referenced by QM-407 must already exist in `theme.css`).
- Epic 9 Phase 0 host-adapter scaffold (`lib/hosts/github/api.js`, `lib/hosts/github/selectors.js`) is merged (commit `2c13564`). All new DOM selectors and HTTP calls in this epic MUST flow through those modules — no direct `document.querySelector` in `content.js` for new code paths, and no inline `fetch()`.
- `lib/update-branch.js` already exposes `updateBranch({ owner, repo, pullNumber, expectedHeadSha, strategy, token })` returning `{ queued: true }` and surfaces `UpdateConflictError` / `UpdateForbiddenError`. Reuse as-is.
- `manifest.json` content_scripts currently match `/pulls*` and `/issues*` only. **PR pages (`/{owner}/{repo}/pull/{n}`) are NOT in scope today** — QM-400 must extend `host_permissions` is unchanged but `matches` MUST add `https://github.com/*/*/pull/*`.

## Parallelizability

Low. ~6 of 11 stories touch the same new PR-page injection code path inside `content.js`. Score: **single-track direct execution**. (See "Workflow decision" below.)

## Goals

- Persistent, warning-styled rebase/update button on every PR page where GitHub exposes the action.
- Confirmation modal before mutation (CI may rerun).
- No duplicate buttons across GitHub soft-nav (Turbo / pjax) transitions.
- No false positives — when rebase is not applicable, hide cleanly.
- Inline approve action and refreshed visual baselines.

## Non-goals

- Auto-rebase on PR pages (that's QM-110 / `lib/auto-rebase.js`, list-page only).
- Replacing GitHub's native merge button.
- Cross-host (GitLab) port — Epic 11.
- Mobile / responsive PR-page layout tuning.

## Execution strategy

**Single-track, single PR, single author.** Per `~/.claude/CLAUDE.md` "Implementation Strategy Default": dispatch parallel only for 4+ truly independent tracks. Here, file-touch ownership is dominated by one new module + one shared file:

| Track | Files owned | Stories |
|-------|-------------|---------|
| **A — PR-page injection (serial)** | `content.js` (new `ensurePrPageRebaseButton` block), `lib/hosts/github/selectors.js` (new selectors), `theme.css` (new component classes), new `lib/qm-pr-page-actions.js` if module grows past ~120 LOC | QM-400, QM-402, QM-403, QM-405, QM-406, QM-408 |
| **B — API extension** | `lib/hosts/github/api.js` (extract `fetchPrState` from `content.js`, add `mergeStateStatus` field if needed) | QM-401 |
| **C — Wire-up (depends on A+B)** | `content.js` glue | QM-404 |
| **D — Tokens** | `theme.css` action-color tokens | QM-407 |
| **E — Tests** | `test/e2e/pr-page-rebase.spec.ts`, `test/e2e/visual/pr-page.spec.ts`, unit tests for new selectors / fetchPrState extraction | QM-409, QM-410 |

Track A stories have hard ordering (detection → render → confirm → soft-nav → fallback). Splitting them into separate sub-PRs would force every follow-up to rebase the previous one's diff in `content.js` — pure cherry-pick churn. **Land them in one branch, internal commit-by-commit ordering preserved for reviewability.**

Tracks B, D, E can technically be split into separate sub-PRs if a contributor wants to land them ahead, but for a 3-day epic with one author the squash overhead beats the parallelism win.

## Risk register

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Soft-nav re-injection produces duplicate buttons | Med | `data-gps-visible-rebase` idempotency guard (per dom_injection.md); e2e test in QM-409 covers two-PR soft-nav. |
| False-positive rebase visibility (button shown when GitHub disagrees) | Med | QM-406 matrix table; never infer from `behind_by` alone — also require native update-branch control present OR `mergeable_state === 'behind'`. |
| GitHub DOM drift breaks merge-panel selectors | High over time | All selectors centralized in `lib/hosts/github/selectors.js`; QM-406 fallback to "Open merge panel" link degrades gracefully when selector fails. Add Sentry breadcrumb on selector miss. |
| Idempotency guard collisions with future features | Low | Use `data-gps-visible-rebase` (specific) not `data-gps-actions` (already used by row injection per dom_injection.md). |
| `manifest.json` matches expansion triggers extension store re-review | Med | Expected; document in CHANGELOG; AMO reviewer note about new pattern + read-only data flow. |
| `expected_head_oid` race when user pushes between fetch and update | Low | `updateBranch` already throws `UpdateConflictError` on 422 — surface "Branch changed, retry" toast and refresh state. |

## Workflow decision: single PR, squash-merge

**Recommendation: ONE PR, squash-merge, single author.**

Rationale:
1. **File-touch overlap is concentrated.** QM-402, QM-403, QM-404, QM-405, QM-406, QM-408 all mutate the same new injection block in `content.js`. Splitting into 6 PRs would force 5 rebases-with-conflicts.
2. **Total scope is small (~3 eng-days).** Below the threshold where parallelism beats integration overhead per `dispatch-or-direct` heuristics.
3. **Hard internal ordering.** Detection (400) → API (401) → render (402) → modal (403) → wire (404) → soft-nav (405) → fallback (406) is a chain — each story validates the previous. Reviewer can read commits in order.
4. **Visual baselines (410) MUST run last** by definition — same logic as QM-220 in Stage 3.
5. **Adversarial review gate runs once** at the end (touches new mutation API surface — worth a Codex pass).

The only candidate for a sub-PR split is **QM-401** (extracting `fetchPrState` to host adapter) — it's a pure refactor, zero behavior change, could land first as a "prep" PR. Author's call. Default: keep in same branch as commit 1.

## Acceptance gate

Before `gh pr ready`:

1. Local green: `npm test && npm run typecheck && npm run lint` — all pass.
2. `npx playwright test` green including new `pr-page-rebase.spec.ts` and refreshed `pr-page.spec.ts` baselines.
3. Coverage ≥ 95% lines (current floor from Stage 3).
4. **Manual smoke against real github.com PR page:**
   - Logged-in user, PR with `behind_by > 0`: button renders yellow, modal confirms, branch updates, toast shows, button hides on refresh.
   - Logged-in user, PR with `mergeable_state === 'clean'`, `behind_by === 0`: no button rendered.
   - Logged-in user, PR without write perms: disabled fallback + "Open merge panel" link.
   - Logged-out user: content script does not error; no button (no token).
   - Soft-nav from PR #N to PR #M: exactly one button each, no orphans.
   - Dark mode: tokens render correctly, modal contrast passes WCAG AA.
5. `codex-adversarial-gate` clean (touches mutation API + new permission scope — required).
6. CHANGELOG entry mentions new `/*/pull/*` content_script match for AMO reviewers.
